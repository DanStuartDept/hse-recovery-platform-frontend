# Internationalisation (i18n) Design — `@repo/i18n` Package

**Date:** 2026-04-08
**Backlog item:** #17 — Internationalisation (i18n)
**Status:** Approved

---

## Goal

Add a lightweight, custom `@repo/i18n` package that provides locale-aware routing, dictionary loading, and translation utilities for the monorepo. No external i18n framework (no `next-intl`, `react-i18next`, etc.) — browser `Intl` APIs are used directly for date/number/list formatting. The package handles the things `Intl` cannot: middleware routing, dictionary management, pluralization in translations, rich text embedding, and a React provider for client components.

## Scope

- New shared package: `@repo/i18n` (source-only, no build step)
- Middleware factory for locale detection and URL rewriting/redirecting
- Dictionary loading with shared + app layers, flat JSON → nested object conversion
- String interpolation, pluralization, and rich text helpers
- `DictionaryProvider` / `useDictionary()` for client components
- App-level integration in `hse-multisite-template` (route restructure, middleware, provider)

### Out of scope

- Wagtail translation API integration (future backend work)
- CMS content translation (CMS pages are translated in Wagtail, not in frontend dictionaries)
- RTL layout support (Irish and English are both LTR)
- Wrapping browser `Intl` APIs (date, number, list, relative time formatting) — apps use these directly with the current locale

### Package: `@repo/i18n`

**Location:** `packages/i18n/`

**Tooling:** `@repo/biome-config`, `@repo/vitest-config`, `@repo/typescript-config`. TSDoc comments on all exported functions, types, and props.
**Documentation:** `README.md` covering package purpose, installation, configuration, all exports with usage examples, dictionary format, and app integration guide.

## Locale Configuration

Each app defines its own locale config. The package provides the type, not the values.

### `I18nConfig` type

```ts
/** Configuration for an app's supported locales. */
export type I18nConfig = {
  /** The default locale — hidden from the URL. */
  defaultLocale: string;
  /** All supported locales, including the default. */
  locales: readonly string[];
};
```

### App-level usage

```ts
// src/lib/i18n/config.ts
import type { I18nConfig } from "@repo/i18n";

export const i18nConfig: I18nConfig = {
  defaultLocale: "en-ie",
  locales: ["en-ie", "ga"],
} as const;
```

Locale list is extensible per-app — not a fixed enum. An app with only `en-ie` is valid (single-locale, no prefix needed). An app needing Ukrainian and Polish adds them to the array.

Config lives at app level, not in `@repo/app-config`, because available locales are a build-time decision per app, not an environment variable.

## Middleware

### `createI18nMiddleware(config: I18nConfig)`

Factory function returning a Next.js middleware function. Each app calls it with its own config.

### Behaviour

| Request                                        | Locales > 1                                            | Single locale                        |
| ---------------------------------------------- | ------------------------------------------------------ | ------------------------------------ |
| `/about/` (no locale prefix)                   | Rewrite to `/{defaultLocale}/about/` (URL stays clean) | Rewrite to `/{defaultLocale}/about/` |
| `/{defaultLocale}/about/` (explicit default)   | Redirect to `/about/` (enforce canonical URL)          | Redirect to `/about/`                |
| `/{otherLocale}/about/` (non-default)          | Pass through                                           | N/A                                  |
| `/about/` with Accept-Language preferring `ga` | Redirect to `/ga/about/`                               | N/A (only one locale)                |

### Locale detection priority

1. `Accept-Language` header (via `negotiator` + `@formatjs/intl-localematcher`)
2. Default locale (fallback)

The proxy is stateless — no cookies are set or read. The URL path is the single source of truth for the active locale. Users switch languages by navigating to the locale-prefixed URL.

### Matcher

Excludes static assets, API routes, and Next.js internals:

```ts
export const config = {
  matcher: ["/((?!api|_next/static|_next/image|assets|favicon.ico|sitemap|robots.txt).*)"],
};
```

The matcher is defined by each app in its `middleware.ts` (not inside the factory) since apps may need to customise it.

### App-level middleware

```ts
// src/middleware.ts
import { createI18nMiddleware } from "@repo/i18n";
import { i18nConfig } from "@/lib/i18n/config";

export const middleware = createI18nMiddleware(i18nConfig);

export const config = {
  matcher: ["/((?!api|_next/static|_next/image|assets|favicon.ico|sitemap|robots.txt).*)"],
};
```

## Dictionaries

### Format

Flat JSON with period-delimited keys. Translator-friendly, compatible with a future Wagtail key-value translation API.

```json
{
  "common.skipToContent": "Skip to main content",
  "common.backToTop": "Back to top",
  "error.heading": "Something went wrong",
  "error.tryAgain": "Try again",
  "notFound.heading": "Page not found"
}
```

### Storage — two layers

| Layer  | Location                                           | Contains                               |
| ------ | -------------------------------------------------- | -------------------------------------- |
| Shared | `packages/i18n/dictionaries/en-ie.json`, `ga.json` | Common UI strings used across all apps |
| App    | `src/dictionaries/en-ie.json`, `ga.json`           | App-specific strings                   |

`getDictionary` merges both layers. App strings override shared strings on key collision.

### Unflattening

Flat JSON is converted to a nested object at load time. The JSON stays flat (source of truth for translators), but the runtime API is nested for developer ergonomics.

**Source:** `{ "error.heading": "Something went wrong", "error.tryAgain": "Try again" }`
**Runtime:** `{ error: { heading: "Something went wrong", tryAgain: "Try again" } }`

Access via dot notation: `dict.error.heading`.

### Type safety

The default locale's dictionary is the source of truth for keys. A utility type `Unflatten<T>` converts the flat key types to a nested object type:

```ts
import type enIe from "../dictionaries/en-ie.json";
type FlatDictionary = typeof enIe;
export type Dictionary = Unflatten<FlatDictionary>;
```

TypeScript catches typos: `dict.eror.heading` is a compile error.

If a non-default locale is missing a key, `getDictionary` falls back to the default locale's value for that key.

### Dictionary loaders

Each app defines loader functions (dynamic imports) so the package stays decoupled from file paths:

```ts
// src/lib/i18n/loaders.ts
import type { DictionaryLoaders } from "@repo/i18n";

export const dictionaryLoaders: DictionaryLoaders = {
  "en-ie": () => import("@/dictionaries/en-ie.json"),
  ga: () => import("@/dictionaries/ga.json"),
};
```

The shared dictionaries are loaded internally by `getDictionary` — apps only define loaders for their own dictionaries.

## String Interpolation

### `interpolate(template, values)`

Simple placeholder replacement. Placeholders use `{name}` syntax.

```ts
interpolate("Welcome, {name}", { name: "Dan" });
// → "Welcome, Dan"
```

No format specifiers — for date/number formatting, use browser `Intl` APIs directly and pass the formatted string as a value.

## Pluralization

### Convention

Keys ending in `_zero`, `_one`, `_other` (matching `Intl.PluralRules` categories). Irish also uses `_two`, `_few`, `_many`.

```json
{
  "search.resultCount_zero": "No results found",
  "search.resultCount_one": "1 result found",
  "search.resultCount_other": "{count} results found"
}
```

### Runtime behaviour

During unflattening, plural sibling keys collapse into a function on the nested object:

```ts
dict.search.resultCount(3); // "3 results found"
dict.search.resultCount(1); // "1 result found"
dict.search.resultCount(0); // "No results found"
```

### `plural(template, count, locale)`

Standalone helper (also used internally by the unflatten logic):

1. Uses `new Intl.PluralRules(locale).select(count)` to determine the category (`zero`, `one`, `other`, etc.)
2. Selects the matching template from the plural group
3. Interpolates `{count}` with the actual value

## Rich Text

### Convention

Strings containing `<tag>...</tag>` patterns represent rich text with embeddable components.

```json
{
  "help.contact": "Contact <link>your local health office</link> for help."
}
```

### `rich(template, tags)`

Maps tag names to React component factories:

```tsx
rich(dict.help.contact, {
  link: (text) => <a href="/offices">{text}</a>,
});
// → ["Contact ", <a href="/offices">your local health office</a>, " for help."]
```

Returns `ReactNode[]` — can be rendered directly in JSX.

Supports multiple tags in a single string. Tags do not nest (no `<a><b>...</b></a>`). Unmatched tags are rendered as plain text.

## React Provider

### `DictionaryProvider`

Server Component in `[lang]/layout.tsx` loads the dictionary and passes it to the provider. Client components access it via hook.

```tsx
// [lang]/layout.tsx (Server Component)
const dict = await getDictionary(lang, dictionaryLoaders);

return <DictionaryProvider dictionary={dict}>{children}</DictionaryProvider>;
```

### `useDictionary()`

Hook for client components. Returns the full typed dictionary object.

```tsx
"use client";
import { useDictionary } from "@repo/i18n";

function TryAgainButton({ onRetry }: { onRetry: () => void }) {
  const dict = useDictionary();
  return <button onClick={onRetry}>{dict.error.tryAgain}</button>;
}
```

## App-Level Integration (`hse-multisite-template`)

### Route restructure

Move all routes under `[lang]/`:

```
src/app/
  [lang]/
    layout.tsx         ← locale-aware layout
    page.tsx
    not-found.tsx
    error.tsx
  global-error.tsx     ← stays at root (outside layout)
```

### Layout changes

`[lang]/layout.tsx` becomes the locale-aware layout:

- Receives `params: Promise<{ lang: string }>` (Next.js 16 — params is a Promise)
- Sets `<html lang={lang}>` dynamically
- Loads dictionary via `getDictionary`
- Wraps children in `DictionaryProvider`

### Next.js config

Add `trailingSlash: true` to `next.config.ts`.

### Error pages

`not-found.tsx`, `error.tsx` — use dictionary strings instead of hardcoded English. `global-error.tsx` stays at root and keeps hardcoded strings (it renders outside the layout and provider).

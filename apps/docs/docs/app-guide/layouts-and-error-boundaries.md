# Layouts and Error Boundaries

## Root layout (`[lang]/layout.tsx`)

The root layout renders the full HTML shell for every page in the app. It runs as a Server Component.

### Responsibilities

**1. Locale validation**

The layout receives `lang` from the URL segment and validates it against the configured locales. If the locale is not in `i18nConfig.locales`, it calls `notFound()` immediately — this prevents the rest of the layout from rendering with an invalid locale.

**2. Dictionary loading**

`loadDictionary` (from `@repo/i18n`) fetches the JSON translation file for the active locale, with the default locale (`en`) as a fallback. If loading fails, the error is logged and re-thrown — a missing dictionary is considered fatal for the layout.

**3. CMS header and footer**

Header and footer data are fetched in parallel from the Wagtail API:

```ts
const [headerResponse, footerResponse] = await Promise.all([
  cmsClient.fetchHeader({ next: { revalidate: REVALIDATE_SECONDS } }),
  cmsClient.fetchFooter({ next: { revalidate: REVALIDATE_SECONDS } }),
]);
```

Both use a 1-hour ISR revalidation interval. If either fetch fails, it logs a warning and passes `null` to the component — the page renders without navigation rather than failing entirely.

**4. Analytics scripts**

Three script components are conditionally rendered based on `@repo/app-config` configuration. See [Analytics Integration](./analytics-integration.md) for details. Their placement in the HTML tree matters:

- `<GtmScripts />` is placed inside `<html>` but before `<body>` — `@next/third-parties` handles the `<script>` and `<noscript>` placement automatically.
- `<OneTrustScripts />` is placed at the top of `<body>` with `strategy="beforeInteractive"` so the consent banner loads before any other scripts.
- `<PiwikProScripts>` wraps the rest of the body content as a React provider.

**5. i18n provider**

`<DictionaryProvider>` (from `@repo/i18n`) makes the flat dictionary and locale available to all client components via context:

```tsx
<DictionaryProvider flat={flat} locale={lang}>
  <SiteHeader data={headerData} />
  {props.children}
  <SiteFooter data={footerData} />
</DictionaryProvider>
```

**6. Static metadata**

The layout exports a `metadata` constant with app-level branding defaults:

```ts
const SITE_NAME = "HSE.ie";
const TITLE_TEMPLATE = `%s | ${SITE_NAME}`;

export const metadata: Metadata = {
  title: { template: TITLE_TEMPLATE, default: SITE_NAME },
  metadataBase: new URL(config.siteUrl),
  formatDetection: { telephone: false },
  icons: { ... },
};
```

These are merged with per-page metadata generated by `generateMetadata` in the catch-all route. The title template means a page with `title: "About"` produces `"About | HSE.ie"` in the `<title>` tag.

## Error boundary (`[lang]/error.tsx`)

A client component (`"use client"`) that Next.js renders when an unhandled error is thrown within the `[lang]` route segment. It has access to the i18n dictionary via `useDictionary<Dictionary>()` because it sits inside the layout's `DictionaryProvider`.

The component:
- Logs the error via `@repo/logger` in a `useEffect` (runs once on mount)
- Renders a translated heading, body text, and a "Try again" button
- Calls `unstable_retry()` when the button is clicked to re-attempt rendering the failed segment

The `error` prop includes an optional `digest` property — a server-side hash of the error that appears in server logs and can be used to correlate a user-reported error to a specific log entry.

## Not-found page (`[lang]/not-found.tsx`)

A client component rendered when `notFound()` is called anywhere within the `[lang]` route — most commonly from the catch-all page when the CMS returns a not-found response.

The page uses `useDictionary<Dictionary>()` for all translatable strings and renders a structured 404 message with links to popular HSE sections (conditions, staff, jobs, about) and a contact link using the `rich()` helper from `@repo/i18n` for inline HTML link interpolation.

## Global error boundary (`global-error.tsx`)

A client component at the root of the app tree. It catches errors that escape the `[lang]` layout — for example, an error thrown during the layout's own execution (dictionary loading failure after the re-throw, or an unexpected crash in the layout Server Component).

Because `global-error.tsx` replaces the entire document when it activates, it must render its own `<html>` and `<body>` tags. It imports the HSE design system SCSS directly (`@hseireland/hse-frontend/packages/hse.scss`) since the normal layout stylesheet is unavailable.

The component uses hardcoded English strings — it cannot use the dictionary because dictionary loading may itself have failed. Like `error.tsx`, it logs the error and provides an `unstable_retry()` button.
